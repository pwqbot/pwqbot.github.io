ucore - Lab5 用户进程管理

发表于2021-05-17|更新于2021-09-20|CS0804n操作系统ucore
|阅读量:

实验内容

在lab4 的基础上增加用户态进程的实现.

需要理解

  • ELF 格式文件.
  • 系统调用的过程
  • 用户态和内核态之间的切换过程
  • fork 的函数逻辑

前置

用户态系统调用

  1. 在用户态调用 syscall

    该函数传入一个系统调用号和可变参数列表, 然后通过 int 命令引发中断.

    一般不直接调用 syscall, 而是会调用被封装过的系统调用函数.

  2. 引起中断之后 cpu 从用户态切换到内核态, 发生了特权级的转换. 然后做以下的事情

    1. cpu 的特权级切换到内核态.

    2. 把所用堆栈切换到内核堆栈.

    3. 压入cpu态的 cs, eip 寄存器. 切换cs , eip 寄存器到相应的中断服务例程.

  3. 跳转到系统调用门, 执行中断服务例程.

    IDT初始化的时候增加了一个系统调用门, 并将其特权级设置为用户态.

    推入中断号和错误码, 跳转到__alltraps.

  4. 在 __alltraps 中把其余所需寄存器推入内核栈中. 并切换段寄存器. 随后调用trap 函数继续处理中断.

  5. trap 函数根据中断发生在进程初始化前后来处理中断.

    为了应对中断处理的过程中又产生中断的情况, 设置 otf 指针指向旧的中断栈顶来维护一个trapframe chain.

    调用trap_dispatch 函数根据中断的类型继续处理中断.

  6. 根据传入的中断号T_SYSCALL, trap_dispatch 函数继续调用内核空间的 syscall 函数执行系统调用

  7. 内核空间的syscall 函数根据用户空间的syscall 函数传入的参数相应对应的系统调用

  8. syscalls 是一个函数数组. 可以通过系统调用号方便的调用相应的函数.

  9. 最后终于来到了系统调用执行的任务入口开始执行任务.

  10. 执行结束之后一路返回到第5步的 trap 函数, 它会判断如果系统调用发生在用户态, 则可能需要调度进程.

  11. 从trap 返回之后再回到__alltraps. 把原来压入内核栈作为中断帧的参数都弹出来.

  12. 使用iret 指令, 弹出eflags, cs, eip 等寄存器的值.

  13. 切换堆栈到用户栈, 特权级转换为用户态.

  14. 继续执行中断发生前的 cs:eip.

过程

用户进程创建

和 lab4 一样, proc_init 函数创建了两个内核线程 idleproc 和 initproc. 但后者要执行的函数 init_main 和之前是不同的.

  • proc_init

initproc 执行 init_main 函数, 它创建了第三个内核进程user_main.

  • init_main

user_main 进程只是用来过渡的(实际上连名字都没给它设置), 它一开始是内核进程, 执行完 user_main 函数, 载入了ELF文件之后就成了第一个用户进程.

  • user_main

KERNEL_EXECVE 是一个宏定义. 它根据传入的名字, 获取对应的要执行的ELF文件的位置和大小. 并把参数传入 kernel_execve 中.

  • KERNEL_EXECVE

kernel_execve 执行系统调用, 注意这里的系统调用不涉及特权级的转换, 一直在内核态运行.

  • kernel_execve

    使用了 int 指令进行中断调用. (不会真有人看不懂内联汇编吧?)

中断调用过程

__alltraps->trap->trap_dispatch->syscall->sys_exec->do_execve->load_icode

最后再从 __alltraps 返回. 在load_icode 中把 tf->eip 改为ELF文件的入口代码. 所以返回之后就开始执行ELF文件.

其他部分不再赘述, 让我们从 sys_exec 开始看.

  • sys_exec

    把传入中断的参数传到do_execve 中并调用

  • do_execve

    1. 设置进程名字

    2. 清空原来的虚拟内存空间.

      载入要执行的ELF文件时会把原来的内存空间覆盖掉.

    3. 调用 load_icode 载入ELF文件

装载ELF文件

ELF(Executable Linkable Format)是linux下可执行文件的格式. 强烈建议阅读一下 <程序员的自我修养> 这本书, 了解一下链接装载的过程.

ELF有4类.

  1. Relocatable File.

    编译完还没链接的文件, 后缀是 .o

  2. Executable File.

    链接完的文件, 可以直接执行, 一般没有扩展名.

  3. Shared Object File.

  4. Core Dump File.

我们要装载的就是ELF的可执行文件Executable FIle.

可以参考一下维基百科这张图. ELF文件可以分成以下几块.

  1. ELF HEADER.

    里面包含了该文件的基本信息和文件结构.

  2. Program Header Table. (装载部分)

    只有可执行的ELF文件会有这部分内容. 在装载的时候为了节约空间不会一个 Section一个 Section 地装载, 而是会把相同权限的段, 把它们合并到一起进行页映射. Program Header Table 就是用来描述这些段的信息, 包括段的权限, 大小, 起始的虚拟位置等等.

  3. Sections.

    这个是ELF文件的主体. 包含代码, 全局变量静态变量, 符号表, 重定位表等信息.

    有 .text section, 里面装代码; .data section, 里面是全局变量和静态变量; .symtab 符号表记录了目标文件定义和引用的全局变量.

  4. Section Header Table.

    描述各个Section结构的.

大致了解了ELF文件后, 接下来进行ELF文件的装载.

在ELF的代码段和数据段中, 已经指定好了要装载到虚拟地址. 我们要做的就是根据它指定好的虚拟地址把它装载进来就好了.

除了装载ELF, 我们还要在装载之后执行该程序, 需要设置 trapframe 的eip为ELF文件的入口程序e_entry.

拷贝内存空间(练习2)

lab5 更新了 copy_mm 函数. 该函数将父进程的内存内容拷贝给子进程.

  • copy_mm

    1. 如果共享内存空间, 设置一下指针就好
    2. 否则创建新的进程控制块和页目录表, 并拷贝虚拟内存空间

拷贝虚拟内存空间的任务由dup_mmap 函数来完成

  • dup_mmap

    复制虚拟内存空间, 即所有的vma, 并调用 copy_range 函数建立页映射

  • copy_range

    复制范围内的内存空间, 按页建立新的内存空间的页映射.

    1. 根据虚拟地址获得旧的物理页page, 并申请一个新的物理页npage.
    2. 把pgae的内容拷贝到npage上.
    3. 建立 npage 到虚拟地址的映射关系.

fork 的返回值

调用fork 之后, 子进程和父进程会执行几乎相同的代码. 于是我们需要判断 fork 之后是在子进程中运行还是在父进程中运行, 判断的依据就是fork 的返回值.

查看用户空间的 sys_fork . 其返回值在syscall 中可以看到, 是返回时中断栈中 eax 寄存器的值.

  • 从父进程中返回

    父进程在 do_fork 调用成功之后会返回子进程的pid, 然后在内核空间的syscall 中把中断栈的eax 设置为do_fork 的返回值.

    所以fork 从父进程返回时, 返回值为其pid.

  • 从子进程中返回

    子进程返回不会经过上面的步骤, 而是直接返回用户态. 但在do_fork 的copy_thread 函数中, 把中断栈的eax 寄存器设置为了0.

    所以当 fork 从子进程返回时其返回值为0.

用户进程退出

当一个用户进程需要退出时, 它会调用 sys_exit 函数.

  • sys_exit

    该函数传入一个退出码, 并访问系统调用接口.

然后就是通过 int 中断陷入内核态balabala, 最后在内核态中执行do_exit.

ucore - Lab4 内核进程管理

发表于2021-04-28|更新于2021-09-20|CS0804n操作系统ucore
|阅读量:

实验内容

lab3 开启了虚拟内存管理机制之后, 我们可以开始用进程来管理所需要执行的任务了 !

进程和简单的代码, 或者函数的区别是什么呢.

虚拟内存可以实现在一个物理内存空间下装载不同的程序, 让每个程序都认为自己享有了整个物理内存空间.

进程的抽象是为了实现在一个 cpu下同时运行不同的程序, 让每个程序都认为自己享有了整个 cpu 资源.

这里的同时只是一个伪概念, cpu 在一个时刻只能执行一个程序((实际上是一条指令). 所以同时指的是 cpu 执行一会任务 A, 然后再执行一会任务 B.

在执行一个程序的时候, 除了要执行的代码, 还要有 寄存器, 堆栈, 内存 这些状态.

于是切换执行的程序的时候, 自然要保存当前执行的状态, 载入下一个程序上次执行的状态.

进程 = 要执行的程序代码 + 执行时的状态.

ucore 本身也是一个大进程.

在本次实验中, 只涉及到了内核进程的管理. 内核进程之间贡献内核虚拟内存空间, 所以创建的时候不需要进行虚拟内存空间的设置.

以下为实验主要任务.

  • 分配并初始化一个进程控制块
  • 为新创建的内核进程分配资源
  • 阅读代码, 理解 proc_run 函数和它调用的函数如何完成进程切换的

主要就是两块, 一个是进程的创建初始化, 还有一个是进程的切换.

前置

直接来到kern_init, 发现增加了一个 proc_init 函数, 来进行内核进程的初始化任务.

proc_init进程初始化做了三件事情.

  1. 初始化进程管理链表和进程哈希表.

  2. 创建并初始化第一个内核进程 idleproc.

  3. 创建并初始化第二个内核进程 initproc.

进程控制块

进程控制块负责进程资源和状态的管理. 在 kern/process/proc.h 中定义了进程控制块的结构.

哈希函数

为了用进程的 pid 快速找到对应的进程, ucore 实现了一个简单的拉链哈希表, 即用链表解决哈希冲突.

  • 哈希函数

    在 libs/hash.c 中定义了一个哈希函数 hash32.

    该函数传入两个参数 val 和 bits, 并将 val 映射到 [0,2bits−1][0,2^{bits} - 1][0,2bits−1] 的范围中.

  • 插入哈希表

    借助该哈希函数定义了一个哈希表 hash_list. 内部嵌套一个双向链表来解决哈希冲突.

  • 根据pid 找到对应的 proc 控制块

    这样就可以用 pid 快速的找到对应的 proc 了.

中断调用

本次实验设计到了中断栈的操作, 所以复习一下中断调用

执行中断

  1. CPU在执行完当前程序的每一条指令后, 都会去确认在执行刚才的指令过程中中断控制器(如:8259A)是否发送中断请求过来, 如果有那么CPU就会在相应的时钟脉冲到来时从总线上读取中断请求对应的中断向量;
  2. CPU根据得到的中断向量(以此为索引)到IDT中找到该向量对应的中断描述符, 中断描述符里保存着中断服务例程的段选择子;
  3. CPU使用IDT查到的中断服务例程的段选择子从GDT中取得相应的段描述符, 段描述符里保存了中断服务例程的段基址和属性信息,此时CPU就得到了中断服务例程的起始地址, 并跳转到该地址;
  4. CPU会根据CPL和中断服务例程的段描述符的DPL信息确认是否发生了特权级的转换. 比如当前程序正运行在用户态, 而中断程序是运行在内核态的, 则意味着发生了特权级的转换, 这时CPU会从当前程序的 TSS 信息(该信息在内存中的起始地址存在 TR 寄存器中)里取得该程序的内核栈地址, 即包括内核态的 ss 和 esp 的值, 并立即将系统当前使用的栈切换成新的内核栈.这个栈就是即将运行的中断服务程序要使用的栈。紧接着就将当前程序使用的用户态的ss和esp压到新的内核栈中保存起来;
  5. CPU需要开始保存当前被打断的程序的现场(即一些寄存器的值), 以便于将来恢复被打断的程序继续执行. 这需要利用内核栈来保存相关现场信息，即依次压入当前被打断程序使用的 eflags, cs, eip, errorCode (如果是有错误码的异常)信息;
  6. CPU利用中断服务例程的段描述符将其第一条指令的地址加载到 cs 和 eip 寄存器中, 开始执行中断服务例程.这意味着先前的程序被暂停执行, 中断服务程序正式开始工作.

返回

  1. 程序执行 iret 指令时, 首先会从内核栈里弹出先前保存的被打断的程序的现场信息, 即 eflags, cs , eip重新开始执行;

  2. 如果存在特权级转换(从内核态转换到用户态), 则还需要从内核栈中弹出用户态栈的ss和 esp, 这样也意味着栈也被切换回原先使用的用户态的栈了;

  3. 如果此次处理的是带有错误码 (errorCode) 的异常, CPU在恢复先前程序的现场时, 并不会弹出 errorCode. 这一步需要通过软件完成，即要求相关的中断服务例程在调用iret返回之前添加出栈代码主动弹出errorCode.

过程

可以把本次实验分成两块, 一个是内核进程的创建和初始化, 一个是内核进程的切换和执行.

创建进程

内核进程的创建由 kernel_thread 函数来实现. 它有三个参数.

  1. 函数指针 fn

    创建的内核进程需要执行的函数.

  2. 参数列表 arg

    fn 的参数.

  3. clone_flags

在kernel_thread函数体中, 创建了一个临时的 trapframe. 设置了

  1. 段寄存器.
  2. 传入的函数和参数放到了里面的 ebx 和 edx 寄存器中.
  3. eip 指向kernel_thread_entry 函数.

之所以要设置这些参数, 是因为在第一次执行内核线程的时候是借助中断返回的过程来跳转到所要执行的函数. 在进程执行的部分会详细解释.

做好了准备工作之后开始真正的进程创建. 这里把 do_fork 和 kernel_thread 分开是为了以后用户进程的创建.

alloc_proc 分配并初始化了一个进程控制块

它的实现也比较简单, 调用 kmalloc 获得控制块的内存空间后把里面的内容都初始化为空即可.

注意这里的页目录表默认设置为内核的页目录表, 如果创建进程之后不更改页目录表的话进程就会共享内核虚拟空间.

2. 分配内核栈

调用 setup_kstack 函数,分配给proc->kstack 一个空的内核栈空间. proc->kstack 指向内核栈的虚拟地址.

3. 设置虚拟内存空间

copy_mm 函数设置进程的虚拟内存空间. 在本次 lab 中只创建内核进程, 共享虚拟地址空间, 无需该函数.

在下次 lab 中创建用户进程的时候需要实现该函数.

4. 设置中断栈和上下文

在内核栈顶开辟一个 trapframe. 把一开始在kernel_thread 中设置好的中断栈传进去.

同时设置了 context 也就是进程上下文

5. 把控制块插入哈希表和进程链表中

为了给进程分配一个唯一的 pid, 在获取 pid 时必须关闭中断响应.

6. 唤醒进程

把进程的状态设置为PROC_RUNNABLE , 这样调度程序才可以调度执行该进程.

7. 函数返回

如果进程初始化成功, 返回其pid. 如果失败根据其发生失败的位置, 推出前要先释放一下分配的空间.

  • cpu_idle

schedule 是进程调度函数, 它尝试寻找一个可运行的进程, 如果没找到则默认运行idle_proc.

  • schedule

找到了可运行的进程后就会调用 proc_run 函数来切换到该进程.

  • proc_run

为什么要load_esp0.

在发生中断的时候, 如果产生了特权级转换. 即由用户态转到内核态. CPU 会从当前的TSS信息中取得内核栈地址 ss 和 esp, 并将系统使用的栈转换成新的内核栈.

每个进程, 包括内核进程之间不共用内核栈, 于是切换的时候自然要重新载入 esp0.

switch 函数执行进程上下文的切换.

    ucore 的进程切换只发生在内核态, 所以进程上下文也比较轻量级, 只有一些通用寄存器.

  • 切换上下文

ret 指令之后就完成了内核进程的切换工作.

第一次执行内核进程

当我们第一次运行新创建的进程时, 它并没有上一次退出的位置, 所以初始化的时候直接把eip 设置为它要执行的函数就好了.

吗?

找到进程初始化时的 copy_thread 函数, 发现上下文中的 eip 被设置为了 forkret 函数.

  • copy_thread

也就是说第一次运行内核进程的时候会先跑到 forkret 函数.

  • forkret

    该函数只是简单调用了 forkrets 函数, 并传入当前进程的中断栈(还记得中断栈里有什么吗?).

  • forkrets

    这个函数在 kern/trap 的中断调用里.

    正常的中断调用会运行到 __alltraps 这, 然后从__trapret 的 iret 指令退出.

    __trapret 是中断调用的返回阶段, 它从中断栈中弹出寄存器等值, 跳转到中断栈里的eip 的位置.

    关于iret 命令

    每个中断服务例程在有中断处理工作完成后需要通过iret 或 iretd指令恢复被打断的程序的执行.

    CPU执行IRET指令的具体过程如下：

    • 程序执行这条iret指令时，首先会从内核栈里弹出先前保存的被打断的程序的现场信息，即 eflags,cs,eip 重新开始执行;
    • 如果存在特权级转换(从内核态转换到用户态), 则还需要从内核栈中弹出用户态栈的 ss 和 esp,即栈也被切换回原先使用的用户栈.(这里内核栈切换没有特权级转换, 所以ss和esp没有用)
    • 如果此次处理的是带有错误码(errorCode)的异常, CPU在恢复先前程序的现场时, 并不会弹出errorCode, 需要要求相关的中断服务例程在调用iret返回之前添加出栈代码主动弹出errorCode.(所以esp跳过了错误码)

forkrets 借用了中断调用的__trapret 函数. 还记得一开始在 kernel_thread 中把要调用的函数和参数放到了中断栈中吗.

  • kernel_thread 和 copy_thread 函数设置了进程的初始中断栈.

于是__trapret 执行完, iret 返回到kernel_thread_entry 函数.

在该函数中, 终于开始运行创建的内核进程真正要运行的函数了.

  • kernel_thread_entry

运行之后调用一个简单的do_exit 返回. 完整的实现在下一个 lab 中

  • do_exit

总结

  1. 内核进程切换只需两个函数 proc_run 和 switch_to 函数即可.

    前者切换 current 指针, 页目录表和 TSS 的 esp0 的值. 后者保存当前上下文, 恢复旧的上下文.

  2. 内核进程的第一次执行绕了点路. 在创建新的内核进程时把 context 中的 eip 设置为中断返回函数. 并将进程所要执行的内容载入中断栈中. 借助中断程序, 弹出中断栈中保存的 eip 等寄存器的值. 最后在 kernel_thread_entry 中跳转到所要执行的函数.

ucore - Lab2 物理内存管理

发表于2021-04-13|更新于2021-05-02|CS0804n操作系统实验
|阅读量:

实验内容

本次实验包含三个部分.

  1. 了解如何发现系统中的物理内存.

  2. 了解如何建立对物理内存的初步管理,即了解连续物理内存管理.

  3. 了解页表相关的操作, 即如何建立页表来实现虚拟内存到物理内存之间的映射, 对段页式内存管理机制有一个比较全面的了解.

  4. 本实验里面实现的内存管理还是非常基本的, 并没有涉及到对实际机器的优化, 比如针对 cache 的优化等. 如果大家有余力, 尝试完成扩展练习.

地址映射

感觉 Lab2 最难理解的地方在几次地址映射关系的切换.

首先得理解好段页式映射.

  • lab1

    在 lab1 中, 地址的映射关系非常简单.

    这是因为 lab1 中段描述符的段基地址为0, 以及其并没有启用分页机制.

而在 lab2 增加了一个 ld 文件 boot.ld. 在编译生成 bootblock 和 kernel 的时候分别用boot.ld 和 kernel.ld.

观察这两个文件, 可以看到两者的虚拟地址起始位置不同.

  • 在boot.ld 中, 程序的起始地址为

    并且bootblock 的代码段也是放在物理地址 0x7c00 开始.

    所以在bootasm.S 和 bootmain.c 中, 地址映射关系为.

  • 而在 lab2 的kernel.ld 中, 程序的虚拟起始地址为

    也就是说 kernel 的代码都从 0xC0100000 开始.

    但是bootloader 仍然是把 kernel的 代码放在地址 0x100000 开始.

    为了得到正确的映射关系, 即把逻辑地址减少0xC0000000 来得到物理地址, lab2 中增加了一个汇编文件 entry.S 用来重新载入 gdt 更新段映射机制.

    可以看到段描述符中的段基地址为- KERNBASE = -0xc000000.

    所以重新载入gdt 之后, 地址映射关系为.

  • eip 指针.

    首先明确 eip 指针存的是程序的逻辑地址, 它是要经过映射才得到物理地址的.

    在kern_entry 代码中有一处需要注意. 在重新载入了gdt 之后, 它使用了一个ljmp 指令来更新eip 的值

    这是因为在 kern_entry 函数重新载入gdt 之前, 虚拟地址 = 线性地址 = 物理地址.

    所以ucore 从bootmain.c 的bootmain 函数跳转到kern_entry 时, 使用的是物理地址来跳转.

    这时候eip 的值为0x1?????. 但是在重新载入了gdt 之后, 虚拟地址不等于物理地址了.

    所以这时候eip 的值应该更新为0xc01?????. 所以要通过ljmp指令来更新eip 的值.

之后在pmm_init 过程中一直保持着这样的映射关系, 直到enable_paging 函数调用, 这时候才使能了页机制.

那么这时候地址映射关系是什么呢. 答案在enable_paging 调用的上面几行

看一下第 12 行的函数 boot_map_segment. 该函数建立了分页机制中线性地址到物理地址的映射.

注意在第 18 行 enable_paging 之后, 之前的段映射还是之前的段映射. 所以这时候的地址映射关系为

也就是说虚拟地址比物理地址高了两个KERNBASE.

所以要再通过gdt_init 函数取消段偏移.

但在调用 gdt_init 之前又有一个小细节, ucore 把虚拟地址为 [KERNBASE, KERNBASE + 4m] 对应的物理地址赋值给了虚拟地址为[0,4m] 对应的物理地址上.

这是因为上面所说的在enable_paging 到gdb_init 的过程中地址映射出现的短暂混乱

所以这时候一个0xCxxxxxxx 的虚拟地址通过分段机制减少了0xC000000. 而我们之前只映射了 0xC000000 以上的线性地址 !

而把[KERNBASE, KERNBASE + 4m] 之间的线性地址对应的物理地址赋值给了 [0,4m] 的物理地址之后.

我们在运行内核态代码, 即只在4m之内的代码的时候.

0xCxxxxxxx 的虚拟地址首先通过分段机制减少0xC000000, 变成了一个[0,4m] 之内的线性地址, 然后这个线性地址对应的物理地址就是

[KERNBASE, KERNBASE + 4m] 对应的物理地址, 也就是物理地址[0,4m] ! 这样就得到了正确的映射关系.

也就是说这一段都是为了gdt_init 的正确调用.

言归正传. 在gdt_init 之后终于ucore 的地址映射变成了我们想要的.

也就是说ucore 的段映射只起了一个权限保护的作用, 在地址映射上实际是没用的.

  • 思考

    为什么在使能页机制的过程中会有这么繁琐的过程呢.

    因为在kernel.ld 之中其设置了程序的虚拟地址为0xc0100000 = 0x100000 + KERNBASE .

    所以在虚拟地址到物理地址之间是有一个映射关系

    在使能页机制之前, 我们只能通过段机制来做一个地址的转换.

    在使能页机制之后, 我们就可以通过页机制进行一个地址的转换.

    也就是说中间的几次 gdt 的重载只是为了在使能页机制之前通过段机制做一个地址映射过渡态.

    最后的段基址都是0x0 , 所以 ucore 实际上只使用了页机制做地址映射.

pmm_init 是整个内核初始化的内存管理部分.

它做了以下几件事:

  1. 初始化内存管理器
  2. 初始化并测试物理页分配机制
  3. 初始化并测试二级页表机制
  4. 更新分段机制

你的first fit算法是否有进一步的改进空间

在这里使用了first_in 分配算法, 即找到第一个满足要求的空闲区.

  • 实现first_in 分配算法

    1. 找到第一个大于等于n的空闲区.
    2. 分配其中物理页. 即把它们从空闲链表中删除, 并设置属性为已分配.
    3. 如果空闲区大小大于 n, 需要把第 n + 1页设置成空闲区头
    4. 返回指向第一个空闲页的描述符的指针, 分配失败返回NULL

  • 内存释放算法

    1. 先找到要释放的物理页的后面第一个空闲链表指针 p.
    2. 把[base, base + n - 1] 这部分物理页描述符插入p前面, 即相当于释放了内存
    3. 设置这部分空闲区头的属性, 然后看一下它有没有和前后空闲区连起来, 有的话要做一点修改.

inline function

讨论一下 inline 关键字对于链接的影响. 有如下几个问题.

  1. 内联函数可以提供外部链接性吗.

    我们希望内联函数在调用他的地方展开, 显然只有在编译阶段知道其完整定义的 Translation Unit(预处理后的c文件) 才有可能做到.

  2. 如果函数没有内联展开, 那么它的行为会是什么. 会生成一个普通版本的函数吗, 还是直接忽略该定义呢.

C99

  • static inline

    1. no externel linkage.
    2. maybe a static non inline version.

    也就是说, 如果成功 inline, 那么就展开, 如果没有 inline, 那么就生成一个 static 版本的普通函数.

  • inline

    1. no externel linkage.
    2. will not produce a code
    3. need a externel version, because maybe use that one(if not inline).

    如果成功 inline 则展开.

    否则的话有点类似一个声明, 但是在该 translation unit 中, 这个函数定义仍然存在, 也就是说也不能再定义一个同名函数.

    但是在调用它的时候则会寻找一个 external version 版本的函数, 如果不提供在链接阶段会出错.

  • extern inline

    1. has external linkage, produce a external linkage version
    2. in the definition file may be inline.

    在拥有 externel inline 的定义的文件中, 有可能会内联展开. 并且总会生成一个外部链接版本的普通函数.

“static inline” means “we have to have this function, if you use it, but don’t inline it, then make a static version of it in this compilation unit”. “extern inline” means “I actually have an extern for this function, but if you want to inline it, here’s the inline-version”. - the Linus

用法

  1. 在头文件中使用static inline.

    坏处是如果不能 inline 会生成许多份同样的函数体, 浪费内存.

  2. 在头文件中用 inline 写一个函数定义, 在对应的 .c 文件中使用 extern inline 写同样的定义. (C99 标准)

    这样当可以 inline 的时候就可以成功展开, 在不能 inline 的时候(如使用了函数的地址指针) 就会使用 extern inline 的版本.

  3. 在头文件中用 extern inline, 在对应的 .c 文件中使用 inline. (GNU89 标准)

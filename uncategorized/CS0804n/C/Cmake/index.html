尝试搞懂 cmake 中关于可见度的三个关键字 PRIVATE, PUBLIC, INTERFACE, 发现了这篇文章Modern CMake is like inheritance, 解释的非常清楚, 这里翻译并且总结一下.

Modern CMake = targets + properties

CMake 的两个核心概念 target 和 properties.

  • target

    1. 可执行文件

      add_executable

    2. 库

      add_library

  • properties

    每个 target 都有着它的一些 properties, 用来指明编译或链接的设置. 比如

    • compilation flags
    • linking flags
    • preprocessor flags
    • C/C++ standard
    • include directories

Using (linking with) libraries behaves like inheritance

可以把 target 比做 C++ 中的类, 每个 target 都是一个类, properties 则是 target 的成员.

target 之间可以 “继承”, 在 CMake 中若 target A 链接了 target B

target_link_libraries(targetA targetB)

可以看作 A 继承了 B.

有三种关键字可以设置 property 的可见度, 当其所属的 target 被其他 target 链接时该 property 会被 (继承/不继承) .

  • PRIVATE

    不被继承

  • PUBLIC

    会被继承

  • INTERFACE

    会被继承

INTERFACE 和 PUBLIC 的差别在于声明为 INTERFACE 的 property 在本 target 中是不可用的, 只是起到一个传递的作用.

可以看到这种机制和 C++ 的类成员的继承机制很像.

同样相似的, 在链接的时候, 也可以指定可见度. 就像 C++ 的类在继承的时候一样.

target_link_libraries(<libA [PRIVATE|PUBLIC|INTERFACE] libB)

  • PRIVATE

    所有 libB 的 PUBLIC 和 INTERFACE properties 都会成为 libA 的 PRIVATE properties.

  • PUBLIC

    … 都会变成 PUBLIC properties.

  • INTERFACE

    … 都会变成 INTERFACE.

Example 1: avoiding header dependencies

假设有以下目录结构.

以及以下 include 依赖

  • LibB

    只内部实现中使用了 LibC

  • LibA

    在它的公共头文件中使用了 LibB

  • main

首先是每个 target 的 properties 设置.

include 库将会提供给外部使用, 自然要用 PUBLIC 声明.

再来处理库之间的依赖关系

  1. 可执行文件的链接类型实际上无所谓.
  2. libB 只在内部实现中使用了 libC, 所以声明成 PRIVATE 即可.
  3. libA 在公共头文件中使用了 libB, 所以需要生成成 PUBLIAC, 否则使用了 libA 的代码将找不到 libB 的函数声明.
  4. target_link_libraries 意味着两件事: 在编译阶段使用改 library 的 properties, 以及在链接阶段将其链接进来.

Example 2: defining header-only libraries

如何处理只有一个头文件的库.

假设 catch2 文件夹下只有一个 catch2.hpp 文件.

add_library 中的 INTERFACE 关键字告诉 CMake, catch2 这个 target 不产生二进制文件, 所以该命令也不需要任何源文件.

target_include_directories 中的 INTERFACE 也很好理解, 因为 catch2 这个 target 没有二进制文件需要生成, 所以他只需要把 include 路径提供给链接到它的 target 即可.
